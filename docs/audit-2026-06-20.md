# Cucina Povera Ledger — Deep-Dive Audit

_Generated 2026-06-20 by an 8-lens audit swarm (42 agents). 28 of 37 findings confirmed against code/data._

## Executive Summary

The single most important pattern is a **slug-resolution split-brain in the Italian arc**: when a week-day's `dinner` string is a bare recipe slug, the week table renders the day's authored `estCalories/estProtein` (Challenge.tsx:511) while `toPlannedDinner` (Challenge.tsx:22-27) logs and modal-displays the recipe's lower `perServing`. For 25 Italian days these disagree by 100–410 kcal, so the same row shows two macro numbers and the value *logged into the ledger* is the lower one. This is the real source of the "crazy calories" report — a display/logging inconsistency, not runaway multiplication; no code path multiplies or double-counts macros, and recipe data is clean (max 710 kcal). Two real *quantity* bugs in `scaleQty` (raw 17-digit floats; over-scaled fixed package sizes) corrupt the live ingredient list on any serving change. Cost display is incoherent across three unlabeled bases, "Cook this week" logs $0 for 93% of weeks, and a handful of recipes understate cost 1.7–2.5×. **DB/source are fully in sync — zero seed drift** (829 recipes, 203 week_themes, 32 docs, 29 challenges all match, 0 value mismatches).

## Critical

*None.* No path produces runaway/absurd values or corrupts the zero-waste scoring moat; the "crazy calories" symptom resolves to the High-severity slug/macro inconsistency below.

## High

### H1. Italian bare-slug days: week view shows `estCalories`, but the planner logs the recipe's lower `perServing` (the "crazy calories" inconsistency)
**Location:** `src/ui/Challenge.tsx:511` (renders `d.estCalories/d.estProtein`) vs `:528-532` → `toPlannedDinner` `:22-27`; data in `src/content/weeks/themes.ts` + `week-1-pasta.ts`.
**Evidence:** `toPlannedDinner` does `r = bySlug.get(dinner)` and returns `r.perServing.*`, discarding the day's `est*`. Across all 31 week files there are exactly **25 bare-slug days where `est != perServing`, every one Italian**; all 38 non-Italian bare-slug days satisfy `est == perServing` (the intended invariant). Examples: `spaghetti-con-la-mollica` est 690/22 vs recipe 600/18; `frittata-di-patate` est 700/30 vs 380/17; `fagioli-all-uccelletto` est 690/25 vs 340/16 (Δ350); `frittata-di-pasta` est 730/30 (week-1-pasta.ts:72-74) / 750/32 (themes.ts) vs 500/24; `ribollita` 720/26 vs 480/19; max Δ410 (`scarola-e-fagioli`). The recipe modal (`:699`) and `DinnerCell` (`:176`) also show the lower `perServing`, so a single week row contradicts itself.
**Fix:** Pick one source of truth for slug-backed days — at seed time set each Italian day's `estCalories/estProtein` equal to the linked recipe's `perServing` (Ashkenazi convention), or route the week view through `toPlannedDinner` so displayed == logged. Note: `week-3-eggs.ts`/`week-4-legumes.ts`/`bonus-sicilian.ts` are **not** separate files — those days live in `themes.ts`; only `week-1-pasta.ts` inlines its own days.

### H2. `scaleQty` renders raw 17-digit floats; the `??` rounding fallback is dead code
**Location:** `src/ui/scale.ts:29` — `return formatQuantity(scaled) ?? String(Math.round(scaled * 1000) / 1000);`; called at `Challenge.tsx:736`.
**Evidence:** `formatQuantity` returns a raw float **string** (not null) for values it can't snap to a fraction, so `??` never fires. Reproduced with the project's real modules: `scaleQty("1/2 cup", 1/6)` → `"0.08333333333333333 cup"`; `scaleQty("3 cloves", 1/7)` → `"0.42857142857142855 cloves"`; `scaleQty("2 eggs", 1/7)` → `"0.2857142857142857 eggs"`. Scanning all 829 recipes across household sizes {1,2,5,7,8}, **174 recipes produce ≥1 raw-float line** (966 total ugly combos). `factor = servings/recipe.servings`, so any of the 98 servings-6, 17 servings-5, etc. recipes hit this at common household sizes.
**Fix:** Round `scaled` to a sane precision (nearest 1/8 or 3 decimals) *before* `formatQuantity`, or detect a decimal-looking return and replace it. The `??` cannot help.

### H3. `scaleQty` over-scales fixed package sizes inside parentheses
**Location:** `src/ui/scale.ts:23-31` (`NUMERIC_TOKEN` global replace), called at `Challenge.tsx:736`.
**Evidence:** Every numeric token is scaled, including physically-fixed can/package sizes. Reproduced: `"1 (28 oz) can" ×2` → `"2 (56 oz) can"`; `"1 block (14 oz)" ×2` → `"2 block (28 oz)"`; `"1 (400g)" ×2` → `"2 (800g)"`; `"1 (3 1/2–4 lb)" ×2` → `"2 (7–8 lb)"`. **118 ingredient lines (~92 distinct)** across the library carry a parenthetical package size and corrupt on any serving change. (`"1 (2-inch)"` and `"(about 14 oz)"` descriptors are also wrongly scaled.)
**Fix:** Skip numeric tokens inside `(...)` — split on parentheticals and run `NUMERIC_TOKEN` only on non-parenthetical segments (or strip `()` spans, scale, re-insert). Add a unit test for `"1 (28 oz) can"`.

### H4. `cold-beet-borscht` and `chao-ga` `estCostPerServing` understated below their own ingredient floor
**Location:** `src/content/recipes/cold-beet-borscht.ts:11`; `src/content/recipes/chao-ga.ts:11`.
**Evidence:**
- **cold-beet-borscht:** est 0.6 × 4 = $2.40, but non-pantry costed ingredients sum to $5.50 = **$1.38/serving (2.3×)**; even beets+sour cream+eggs alone = $4.50.
- **chao-ga:** est 1.05 × 4 = $4.20, but non-pantry sum = $7.05 = **$1.76/serving (1.68×)**; the $5.50 whole chicken alone is $1.38/serving, already above est.

These feed display (`Challenge.tsx:177,701`) and the planner's cost-ranking key (`planner.ts:46`). Across all 829 recipes only 5 exceed 1.5× (schav 2.57, borscht 2.40, chao-ga 1.68, omurice 1.61, nopales-con-huevo 1.54) — genuine outliers, not convention.
**Fix:** Raise est to the per-serving non-pantry floor (~1.4–1.5 and ~1.75–2.0 respectively), or correct over-attributed ingredient costs.

## Medium

### M1. "Cook this week" logs `cost: 0` for 189 of 203 weeks
**Location:** `src/ui/Challenge.tsx:528-532` (fallback `cost: 0`) + `toPlannedDinner` `:27`; consumed by `src/core/ledger.ts:78,89,90`.
**Evidence:** `cost: r?.estCostPerServing ?? fallback.cost`, and the fallback is literal `0`. Only 63 of 1421 day-dinners exactly match a recipe slug; grouped into weeks, **14 weeks have ≥1 cost-bearing dinner (all Italian-family), 189 are 100% free-text** → every cooked dinner logged at $0. `summarize()` then reports cost/costPerKcal/costPerProtein as 0 for ~93% of weeks, even though `WeekPlan.steadyStateWeekly` and per-`IngredientLine.cost` exist. (The 0 lands as an overwritable form prefill, hence Medium not High.)
**Fix:** Give `WeekDay` an `estCost` field used as the fallback, or derive a per-day cost from `steadyStateWeekly/7`.

### M2. Curated shopping-list footer total disagrees with the `firstShopTotal` badge (~2× on early weeks)
**Location:** `src/ui/components/ShoppingList.tsx:54` (renders `list.totalCost`) under `Challenge.tsx:517` (`First shop {firstShopTotal}`); `src/core/shopping.ts:135-145,209-224`.
**Evidence:** `totalCost = round2(Σ authored shopping[].cost)` while the badge shows the authored `firstShopTotal`; the two render in the same panel and disagree. e.g. `week-1-pasta.ts`: badge "First shop $32.00" vs footer "$86.50 · fresh $60.50 · pantry $26.00"; bonus-sicilian 97 vs 36; week-3-eggs 75.5 vs 30. **Correction to the original framing:** the mismatch is *not* duplicate lines — week-1-pasta's 19 lines are all distinct single-quantity items, and the badge≠footer gap holds for ~all weeks (only 1/203 reconcile exactly). The right fix is rendering, not re-authoring.
**Fix:** Make the footer display `firstShopTotal`/`steadyStateWeekly` (or reconcile the two). Do **not** "de-duplicate lines" — there are no duplicate lines.

### M3. Side-by-side cost figures use three unlabeled bases
**Location:** `src/ui/Challenge.tsx:317` & `:364` (generated/AI `Week total … · budget $14`), `:517-519` (curated `First shop`/`Steady state`).
**Evidence:** Generated/AI "Week total" = Σ of 7 single-serving `estCostPerServing` (`planner.ts:162`, typical $5–8); curated "First shop" = `firstShopTotal` household batch incl. pantry ($22–36); "Steady state" $13–24/wk; budget a flat $14 (seeded identically for all challenges). Three incompatible bases for "a week," no unit labels. (Minor: the `budget $14` line renders only on generated/AI cards, not beside the curated `firstShopTotal`.)
**Fix:** Label each figure with its basis, or normalize to one basis before comparing to budget.

### M4. Week-day `est*` contradicts linked recipe `perServing` (DB-level restatement of H1)
**Location:** `week_themes.days` vs `recipes.per_serving_*`; rendered `Challenge.tsx:511` vs resolved `:528-532`.
**Evidence:** Same 25 Italian slug-backed days, confirmed equally against the seeded DB columns. This and H1 are the same defect at source and DB layers — fix once.
**Fix:** Reconcile at seed time (set day `est*` = recipe `perServing`) or route the week view through `toPlannedDinner`.

### M5. ~26 shard-A recipes understate `estCostPerServing` vs their non-pantry ingredient floor by >10%
**Location:** `src/content/recipes` (a–m).
**Evidence:** Σ non-`spansWeeks` `IngredientLine.cost` / servings exceeds `estCostPerServing` by >10% in ~26 files (excluding H4): `huevos-en-salsa-verde` 1.38×, `baked-eggs-with-onions` 1.36×, `farmer-cheese-and-noodles` 1.32×, `ginisang-gulay` 1.33×, `huevos-a-la-mexicana` 1.26×, `cheese-blintzes` 1.25×, `butter-beans` 1.21×, `holishkes` 1.21×.
**Fix:** Bump each `estCostPerServing` to ≥ (non-pantry sum / servings), or correct over-attributed ingredient costs.

### M6. Weeks repeat the same dinner on two days (violates "seven distinct dinners")
**Location:** `src/content/weeks/ashkenazi-weeks.ts` (all 7 weeks); `src/content/weeks/themes.ts` (polenta week).
**Evidence:** Exact-string slug repeats within one 7-day week: every Ashkenazi week has one — `potato-latkes` Mon+Fri, `schmaltz-herring` Tue+Fri, `kasha-varnishkes` Mon+Thu, `holishkes` Mon+Thu, `cheese-blintzes` Tue+Fri; week-6 has **two** repeats; `gefilte-fish` Tue+Fri. Polenta: `polenta-e-fagioli` Sun + Wed. Each repeated bare slug resolves to the identical recipe, violating CLAUDE.md rule 6.
**Fix:** Replace the duplicated day with a distinct dinner from that week's pool, or reframe as a leftover with distinct prose/slug.

## Low

### L1. Pantry "bought once" staple cost is summed across every dish it appears in
`src/core/shopping.ts:185-186`. `line.cost += ing.cost ?? 0` runs unconditionally; the pantry flag only affects fresh/pantry classification, never dedup. A staple in all 7 dinners reports ~7× one purchase. `freshCost` is fine; only `pantryCost` over-states up-front spend (non-gating). *Fix:* take the max per-occurrence cost for pantry lines, or document the field as an amortized per-week share.

### L2. Core `validateEntry` accepts non-integer calories/protein the DB and MCP reject
